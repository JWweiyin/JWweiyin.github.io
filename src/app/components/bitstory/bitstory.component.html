About bitStory!


Rhythm games usually tend to be arcade-style and rarely story-based. Because of this, players are inclined to not be very emotionally invested in the game and lose interest quickly. Hence we have created a game with mix of rhythm game and visual novel elements to get players to be invested in our game and will therefore be motivated to continue playing such games.

Most games of the visual novel genre are not very engaging in terms of gameplay, mainly consisting of clicking the same button to advance dialogue. By adding in elements of rhythm games, a skill-based gentre, we made a game with visual novel elements that is engaging and interactive for the player.

My contributions to this team project were:

  • Composition of music for rhythm game levels, level design for rhythm game and programmer for rhythm game
  • Programmer for user interface and menus


Development Process


The game was developed in three iterative stages, or milestones.

Milestone 1 featured:

  • One rhythm game level with the scoring and combo systems implemented.

Milestone 2


Milestone 2 featured:

  • A main menu that allowed the user to choose between Story Mode (the main game mode, consisting of alternating visual novel and rhythm game segments) and Arcade Mode (a 'bite-sized mode' allowing the player to play already cleared levels), or quit the game.
  • A Story Mode with a visual novel prologue, and one rhythm game level.
  • An Arcade Mode with five rhythm game levels for the user to choose from.
  • Results screens upon the conclusion of each rhythm game level.

Feedback was obtained from one test player and the criticism was as follows:

  • Inputs for the rhythm game were occasionally unresponsive. This was later discovered to be an error in the code for the beatmap component, and was amended in the next milestone.
  • The camera angle and perspective of the rhythm game segment may have made it difficult for the user to see where the falling notes overlapped with the buttons. We solved this issue with a design overhaul of the level layout in the next milestone.

Milestone 3


Milestone 3 featured:

  • A level design overhaul, where the size of the buttons were reduced.
  • A fully fledged Story Mode, which incorporated a full plotline and eight rhythm game levels.
  • Implementation of a Pass/Fail system for rhythm game levels. Upon scoring more than 700000 points for a rhythm game level, the user would have been deemed to have passed the level and the level would be available in arcade mode. Otherwise, the level would not be available in arcade mode.
  • Implementation of an automatic save system for story mode, where the player would be able to continue their gameplay from the game segment they last ended their play session on.
  • Pause screens for visual novel and rhythm game segments.
  • An options menu with a volume slider for music and keybinding options.
  • An achievements page where high scores for rhythm game levels could be viewed.
  • A tutorial screen for the visual novel segment.

User Testing

The Github repository link, as well as a feedback form, were sent to a handful of volunteers to gather feedback.
There were a number of aspects which we received feedback on, but did not have the time to adjust in game.

These aspects were:

  1. Smoother transitions between the visual novel and rhythm game segments.
  2. Allowing story progression after failing a level a certain number of times.
  3. Lighting was not consistent across all the rhythm game levels.

Bugs Squashed

  1. Developer and user testing uncovered an error in which when the game was first played, all the buttons in the rhythm game segment were mapped to a single key. This was identified to be an initialization error in the keybinding system, which was amended to fix the issue.
  2. User testing uncovered an error in which when the log showing previous dialogue was opened during the visual novel segment, it would remain there even during the rhythm game. This was identified to be a scripting error where the log and other visual novel UI components were not deactivated before loading a rhythm game segment screen. The code was amended to fix the issue.
  3. User testing uncovered an error in which after restarting a rhythm game segment, the game was paused. This was a scripting error in which the method to resume the game was not called in the reload method. The code was amended to fix the issue.

Problems Encountered

  • User testing uncovered that the visuals/audio of the rhythm game were occasionally laggy, though such delays could arise from the computer not having sufficient memory to run the game smoothly if other programs were run at the same time. This was beyond our control to fix, though it would be recommended to close most programs while running the game.

Learning Points


This being my first attempt at a software engineering project, many aspects could be better streamlined. Many code sections had functional overlaps which made debugging a more complicated process than it should have been. Now with better understanding of software engineering principles and software design patterns, I understand how the different classes within the architecture could have been better organised, and will keep that in mind for the next time I embark on such a project.
